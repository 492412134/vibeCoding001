# 灰度发布功能使用与运维指南

## 1. 功能介绍

灰度发布是一种软件发布策略，允许在生产环境中逐步推出新功能或新版本，通过控制流量比例来降低发布风险。本功能通过 Nacos 配置中心实现灰度规则的动态管理，支持基于多种维度的流量控制。

### 核心特性

- **动态配置**：通过 Nacos 配置中心实时管理灰度规则
- **多维度条件**：支持用户ID、IP地址、请求参数等多种条件匹配
- **权重分配**：支持基于权重的流量分配
- **零停机发布**：无需重启服务即可更新灰度规则
- **版本管理**：基于服务实例的版本标记进行路由

## 2. 配置说明

### 2.1 服务实例配置

所有需要参与灰度发布的服务实例需要在注册到 Nacos 时携带版本元数据。

#### 配置方式

在服务的 `application.yml` 文件中添加以下配置：

```yaml
spring:
  cloud:
    nacos:
      discovery:
        metadata:
          version: ${SERVICE_VERSION:v1}
```

#### 版本设置

- 默认版本为 `v1`
- 通过环境变量 `SERVICE_VERSION` 设置版本，例如：
  ```bash
  SERVICE_VERSION=v2 java -jar payment-service.jar
  ```

### 2.2 灰度规则配置

在 Nacos 控制台中创建 `gray-rules.yaml` 配置文件：

#### 配置信息

- **Data ID**: gray-rules.yaml
- **Group**: DEFAULT\_GROUP
- **配置格式**: YAML

#### 配置示例

```yaml
gray:
  rules:
    # 支付服务灰度规则
    - service: vibe-payment-service
      enabled: true
      weight: 30  # 30%流量进入灰度版本
      conditions:
        # 用户ID条件
        - type: user_id
          operator: in
          values: ["1001", "1002", "1003"]
        # IP地址条件
        - type: ip
          operator: regex
          values: ["192\.168\.1\..*"]
      version: v2
    
    # 订单服务灰度规则
    - service: vibe-order-service
      enabled: false
      weight: 0
      conditions: []
      version: v1
```

### 2.3 条件类型说明

| 条件类型     | 说明    | 示例值                         |
| -------- | ----- | --------------------------- |
| user\_id | 用户ID  | \["1001", "1002"]           |
| ip       | IP地址  | \["192.168.1..\*"]          |
| header   | 请求头   | \["X-App-Version", "1.0.0"] |
| param    | URL参数 | \["version", "v2"]          |

### 2.4 操作符说明

| 操作符         | 说明     | 适用类型 |
| ----------- | ------ | ---- |
| in          | 包含在列表中 | 所有类型 |
| not\_in     | 不在列表中  | 所有类型 |
| equals      | 等于     | 所有类型 |
| not\_equals | 不等于    | 所有类型 |
| regex       | 正则匹配   | 所有类型 |

## 3. 使用方法

### 3.1 启动多版本服务实例

#### 启动默认版本（v1）

```bash
# 支付服务
cd vibe-payment-service
mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Dserver.port=10991"

# 订单服务
cd vibe-order-service
mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Dserver.port=10992"
```

#### 启动灰度版本（v2）

```bash
# 支付服务 v2 版本
cd vibe-payment-service
SERVICE_VERSION=v2 mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Dserver.port=10993"

# 订单服务 v2 版本
cd vibe-order-service
SERVICE_VERSION=v2 mvn spring-boot:run -Dspring-boot.run.jvmArguments="-Dserver.port=10994"
```

### 3.2 配置灰度规则

1. 访问 Nacos 控制台：<http://localhost:8848/nacos>
2. 登录账号：nacos / nacos
3. 进入配置管理 → 配置列表
4. 点击 "+" 按钮添加或编辑 `gray-rules.yaml` 配置
5. 根据需要设置灰度规则

### 3.3 监控灰度效果

#### 查看 Gateway 日志

```bash
# Gateway 日志中会记录灰度路由的情况
2024-01-01 10:00:00.000  INFO 12345 --- [nio-8081-exec-1] c.e.gateway.gray.GrayRouteFilter        : Gray routing enabled for service: vibe-payment-service, version: v2
```

#### 查看服务实例日志

- 观察不同版本实例的请求量
- 确认灰度流量是否正确路由

## 4. 运维操作

### 4.1 灰度发布流程

1. **准备阶段**：
   - 部署新版本服务实例（v2）
   - 确保服务实例正常注册到 Nacos
2. **灰度开启**：
   - 在 Nacos 中配置灰度规则
   - 设置较小的灰度比例（如 10%）
   - 观察系统运行状态
3. **灰度扩大**：
   - 逐步增加灰度比例（如 30% → 50% → 80%）
   - 持续监控系统性能和错误率
4. **全量发布**：
   - 将灰度比例设置为 100%
   - 确认所有流量都路由到新版本
5. **清理阶段**：
   - 下线旧版本服务实例
   - 调整灰度规则或禁用灰度功能

### 4.2 回滚操作

如果在灰度发布过程中发现问题，可立即回滚：

1. **快速回滚**：
   - 在 Nacos 中修改灰度规则，将 `enabled` 设置为 `false`
   - 或将 `weight` 设置为 `0`
2. **彻底回滚**：
   - 停止新版本服务实例
   - 确保所有流量路由到旧版本

### 4.3 常见问题处理

| 问题       | 原因         | 解决方案                                   |
| -------- | ---------- | -------------------------------------- |
| 灰度路由不生效  | Nacos 配置错误 | 检查配置格式和服务名称                            |
| 服务实例无法注册 | 版本元数据未设置   | 检查服务配置中的版本设置                           |
| 配置变更不生效  | 配置刷新未开启    | 确保 Gateway 配置中 `refresh-enabled: true` |
| 灰度流量分配不均 | 权重设置问题     | 调整权重值，确保流量分配符合预期                       |

## 5. 最佳实践

### 5.1 灰度策略建议

- **从小规模开始**：初始灰度比例建议设置为 5-10%
- **逐步扩大**：每次增加 20-30% 的灰度比例
- **监控先行**：在灰度发布前确保监控系统正常运行
- **回滚预案**：制定详细的回滚计划，确保出现问题时能快速响应

### 5.2 配置管理建议

- **版本控制**：对灰度规则配置进行版本管理
- **环境隔离**：不同环境使用不同的 Nacos 命名空间
- **配置备份**：定期备份灰度规则配置
- **权限控制**：限制灰度规则的修改权限

### 5.3 性能优化

- **缓存机制**：利用规则缓存提高匹配性能
- **条件简化**：避免过于复杂的条件组合
- **实例健康检查**：确保灰度实例的健康状态
- **负载均衡**：在灰度实例之间合理分配流量

## 6. 监控与告警

### 6.1 监控指标

- **灰度流量比例**：监控实际灰度流量占比
- **服务实例健康状态**：监控各版本实例的健康情况
- **请求响应时间**：对比不同版本的响应时间
- **错误率**：监控各版本的错误率

### 6.2 告警设置

- **灰度比例异常**：当实际灰度比例与配置不符时告警
- **服务实例异常**：当灰度实例健康状态异常时告警
- **错误率上升**：当灰度版本错误率超过阈值时告警
- **响应时间延长**：当灰度版本响应时间明显延长时告警

## 7. 总结

灰度发布功能为系统升级提供了一种安全、可控的方式，通过精细的流量控制和实时的配置管理，大大降低了发布风险。正确使用灰度发布功能，可以实现零停机的服务升级，提高系统的可靠性和稳定性。

### 关键优势

- **降低风险**：通过逐步放量，及时发现并解决问题
- **快速回滚**：出现问题时可以立即切换回稳定版本
- **数据驱动**：基于实际运行数据做出发布决策
- **用户体验**：减少发布对用户的影响

### 适用场景

- **新功能上线**：先让部分用户体验新功能
- **版本升级**：逐步迁移用户到新版本
- **A/B 测试**：对比不同版本的性能和用户反馈
- **紧急修复**：快速部署修复版本，最小化影响范围
